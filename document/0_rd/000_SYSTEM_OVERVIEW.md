## システム概要

| 項目      | 内容                                                                                                  |
|---------|-----------------------------------------------------------------------------------------------------|
| システム名   | wakasaba_orchestrator                                                                               |
| 目的      | 複数のLinuxサーバに対して定型メンテナンス処理（OS更新・PaperMCサーバ更新・バックアップ等）を自動実行するオーケストレーションバッチ                            |
| 実行形式    | コマンドラインバッチ（Fat JAR）                                                                                 |
| 実行契機    | 手動実行（スケジューラ等からの呼び出しを含む）                                                                             |
| 実行コマンド例 | `java -jar wakasaba_orchestrator-1.0-SNAPSHOT-all.jar <server0_file> <server1_file> <server2_file>` |

---

## 背景・目的

- 複数台のLinuxサーバ（Minecraftサーバを含む）の定期メンテナンスを手動で行うと、作業漏れ・手順ミスのリスクがある。
- 本システムはメンテナンス手順をプログラムとして記述し、一括・自動実行することで作業の確実性と効率化を実現する。
- PaperMC（Minecraftサーバソフトウェア）および関連プラグイン（Pl3xMap）の最新版への自動更新も対象に含める。

---

## 対象サーバ構成

| 識別子  | 接続方式       | 役割                       | 処理概要                                                                     |
|------|------------|--------------------------|--------------------------------------------------------------------------|
| サーバ0 | SSH（リモート）  | 汎用Linuxサーバ               | OSパッケージ更新・即時再起動                                                          |
| サーバ1 | SSH（リモート）  | PaperMCサーバ（Minecraftサーバ） | PaperMC停止・OSパッケージ更新・バックアップ・PaperMC/Pl3xMap最新版取得・配置・即時再起動・起動待機後にPaperMC起動 |
| サーバ2 | SSH（リモート）  | 汎用Linuxサーバ               | OSパッケージ更新・即時再起動                                                          |
| 自サーバ | Bash（ローカル） | 本バッチを実行するサーバ自身           | OSパッケージ更新・60秒遅延バックグラウンド再起動                                               |

## 機能要件一覧

### FR-01：コマンドライン引数の受付・検証

- システムはコマンドライン引数として正確に3つのファイルパスを受け付けること。
- 引数の数が3つでない場合、USAGEメッセージを標準出力へ表示し、処理を行わずに終了すること。

### FR-02：接続情報ファイルの読み込み

- 各引数に指定されたファイルパスから、SSHサーバへの接続情報（ホスト・ポート・ユーザ名・秘密鍵パス）を読み込めること。
- 接続情報ファイルは1行1値のテキストファイル（UTF-8）であり、順番に host / port / user / key を格納していること。

### FR-03：リモートサーバ（サーバ0・サーバ2）のOSメンテナンス

- SSH経由で以下のコマンドを順次実行できること。
    1. `sudo apt update`（パッケージリスト更新）
    2. `sudo apt upgrade -y`（パッケージアップグレード）
    3. `sudo shutdown -r now`（即時再起動）

### FR-04：PaperMCサーバ（サーバ1）のメンテナンス

SSH経由で以下の処理を順次実行できること。

1. PaperMCサービス停止（`sudo systemctl stop papermc`）
2. OSパッケージ更新（`sudo apt update`）
3. OSパッケージアップグレード（`sudo apt upgrade -y`）
4. バックアップシェルスクリプト実行（`sh /home/mini/mcs/shell/backup.sh`）
5. PaperMCおよびPl3xMapの最新版JARのダウンロード（後述FR-05）
6. ダウンロードファイルのSHA検証・本番ディレクトリへの配置（後述FR-06）
7. 即時再起動（`sudo shutdown -r now`）
8. 再起動完了まで自動待機（SSH接続可能になるまでポーリング）
9. PaperMCサービス起動（`sudo systemctl start papermc`）

### FR-05：PaperMC/Pl3xMap最新版の自動取得

- PaperMC公式APIを呼び出し、最新のバージョン文字列を取得できること。
- 取得したバージョン文字列を用いてPaperMCビルド情報APIを呼び出し、最新ビルドのダウンロードURLを取得できること。
- 取得したダウンロードURLを用いてリモートサーバ上でwgetコマンドを実行し、JARファイルをダウンロードできること。
- Modrinth APIを呼び出し、Pl3xMapの最新バージョンのダウンロードURLを取得できること。
- PaperMCバージョンとPl3xMapの対応バージョンが一致しない場合、Pl3xMapのダウンロードをスキップすること。
- ダウンロードしたJARファイルは所定のダウンロード用ディレクトリへ保存すること。

### FR-06：JARファイルのSHAチェックサム検証・配置

- ダウンロードしたPaperMC JARのSHA-256チェックサムを検証し、一致した場合のみ本番ディレクトリへ移動すること。
- ダウンロードしたPl3xMap JARのSHA-512チェックサムを検証し、一致した場合のみ本番ディレクトリへ移動すること。
- Pl3xMapの旧バージョンのJARファイルは、バージョン一致・SHA検証の結果にかかわらず本番プラグインディレクトリから削除すること。
- SHA検証に失敗した場合、ファイル移動をスキップすること（エラーとして停止しない）。

### FR-07：自サーバのOSメンテナンス

- ローカルBashコマンドとして以下の処理を順次実行できること。
    1. `sudo apt update`（パッケージリスト更新）
    2. `sudo apt upgrade -y`（パッケージアップグレード）
    3. 60秒遅延後の再起動（`(sleep 60 && sudo shutdown -r now) &`、バックグラウンド実行）

### FR-08：サーバ再起動後の自動待機・再接続

- すべてのSSHコマンド実行前に、対象サーバへのSSH接続が可能な状態になるまで自動的に待機すること。
- これにより、サーバ再起動後の後続コマンドも再起動完了を待ってから実行される。
- 待機はポーリング（1秒間隔での接続試行）によって実現すること。

### FR-09：処理シーケンスの直列実行

- 各サーバへの処理は直列に実行し、前のサーバの全処理完了後に次のサーバの処理を開始すること。
- 実行順序は サーバ0 → サーバ1 → サーバ2 → 自サーバ とすること。

### FR-10：実行ログの記録・ファイル出力

- バッチ開始時刻・終了時刻をログに記録すること。
- 各サーバへの処理開始時に区切りログを記録すること。
- 実行したコマンドおよびその標準出力をログに記録すること。
- バッチ正常終了後、収集したログをファイルへ出力すること。
- ログファイル名は終了時刻のタイムスタンプ（yyyyMMddHHmmss形式）を含むこと。

## 非機能要件

### NFR-01：信頼性

- 各コマンドの実行・出力はすべてログへ記録し、実行後に内容を確認できること。
- JARファイルの配置前にチェックサム検証を行い、不正なファイルの配置を防止すること。

### NFR-02：保守性

- 実行環境（ファイルパス・コマンド文字列・URL・ログフォーマット等）に依存する値はすべて定数クラス（`WksConstants`
  ）に一元管理し、修正を1箇所に集約できること。
- パッケージ・クラスの責務を明確に分離し、各クラスが単一の責務を持つこと。

### NFR-03：セキュリティ

- SSHの秘密鍵ファイルパスはログへ出力しないこと。
- SSH接続情報ファイルのパーミッションは600（所有者のみ読み書き可）に設定することを推奨すること。

### NFR-04：移植性・拡張性

- Fat JARとして単一ファイルで配布・実行できること。
- 接続情報ファイルを変更することで、対象サーバのIPアドレス・ポート・ユーザ・秘密鍵を変更できること。

## 入力要件

### IN-01：コマンドライン引数

| 引数インデックス | 内容                                    |
|----------|---------------------------------------|
| args[0]  | サーバ0の接続情報ファイルパス（テキストファイルの絶対パスまたは相対パス） |
| args[1]  | サーバ1（PaperMCサーバ）の接続情報ファイルパス           |
| args[2]  | サーバ2の接続情報ファイルパス                       |

### IN-02：接続情報ファイル仕様

1行1値のテキストファイル（UTF-8）。行の順番は以下に固定する。

| 行番号 | 項目   | 型   | 説明                  |
|-----|------|-----|---------------------|
| 1行目 | host | 文字列 | SSH接続先IPアドレスまたはホスト名 |
| 2行目 | port | 整数  | SSH接続ポート番号          |
| 3行目 | user | 文字列 | SSH接続ユーザ名           |
| 4行目 | key  | 文字列 | SSH認証用秘密鍵ファイルの絶対パス  |

## 例外・エラー要件

### ER-01：基本方針

- 業務メソッドで発生した例外は、`throws` 宣言によって上位クラスへ委譲し、最終的に `Main.main` まで伝播させること。
- `Main.main` においても例外のキャッチ・回復処理は行わず、JVMが標準エラーへスタックトレースを出力して異常終了すること。

### ER-02：例外の種類と想定発生状況

| 例外クラス                            | 主な想定発生状況                                         |
|----------------------------------|--------------------------------------------------|
| `IOException`                    | 接続情報ファイル読込失敗・SSH標準出力読取失敗・HTTPネットワーク障害・ログファイル書込失敗 |
| `InterruptedException`           | SSH/HTTP待機中の割り込み・ローカルプロセス待機中の割り込み                |
| `JSchException`                  | SSHセッション生成失敗・SSH接続失敗・チャンネル接続失敗                   |
| `FileNotFoundException`          | ログファイル出力先ディレクトリの不在またはファイル生成失敗                    |
| `NumberFormatException`（非検査）     | 接続情報ファイルのポート番号が整数値でない                            |
| `IndexOutOfBoundsException`（非検査） | 接続情報ファイルが4行未満                                    |

### ER-03：SSH接続復旧の特例

- サーバ再起動後の接続確認ポーリング中に発生する `JSchException`（接続不可）は、サーバが起動中であることを示す正常な状態として
  `false` 扱いで吸収し、ポーリングを継続すること。
- 秘密鍵読み込みエラーなど、接続不可以外の原因による `JSchException` は吸収せず上位へ伝播させること。
